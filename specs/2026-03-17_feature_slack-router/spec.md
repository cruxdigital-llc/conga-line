# Spec: Slack Event Router

## Overview
Build a Node.js router service that maintains one Socket Mode connection to Slack, routes events by channel/user to the correct Conga Line container via internal HTTP, and preserves zero-ingress security. Containers switch from Socket Mode to HTTP webhook mode.

## Deliverables

### 1. Router Application (`router/`)

#### `router/package.json`

```json
{
  "name": "conga-slack-router",
  "version": "1.0.0",
  "private": true,
  "type": "module",
  "main": "src/index.js",
  "scripts": {
    "start": "node src/index.js"
```

#### `router/src/index.js`

```javascript
import { SocketModeClient } from '@slack/socket-mode';
import { readFileSync } from 'fs';

// Load config
const CONFIG_PATH = process.env.ROUTER_CONFIG || '/opt/conga/config/routing.json';
const config = JSON.parse(readFileSync(CONFIG_PATH, 'utf-8'));

const appToken = process.env.SLACK_APP_TOKEN;
if (!appToken) throw new Error('SLACK_APP_TOKEN required');

// Socket Mode client
const client = new SocketModeClient({ appToken });

// Route an event to the correct container
async function routeEvent(body) {
  let target = null;

  // Extract channel from various event shapes
  const channel = body?.event?.channel
    || body?.event?.item?.channel
    || body?.channel?.id
    || body?.channel;

  if (channel) {
    if (channel.startsWith('D')) {
      // DM — route by sender's user ID
      const userId = body?.event?.user || body?.user?.id || body?.user;
      if (userId && config.members[userId]) {
        target = config.members[userId];
      }
    } else {
      // Channel message — route by channel ID
      target = config.channels[channel];
    }
  }

  if (!target) {
    // Try user-based routing as fallback (app_home, etc.)
    const userId = body?.event?.user || body?.user?.id || body?.user;
    if (userId && config.members[userId]) {
      target = config.members[userId];
    }
  }

  if (!target) {
    console.log(`[router] No route for channel=${channel} — dropping event`);
    return;
  }

  try {
    const res = await fetch(target, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(body),
    });
    if (!res.ok) {
      console.error(`[router] Forward failed: ${res.status} → ${target}`);
    }
  } catch (err) {
    console.error(`[router] Forward error to ${target}:`, err.message);
  }
}

// Handle all Socket Mode event types
for (const eventType of ['events_api', 'interactive', 'slash_commands', 'block_suggestion']) {
  client.on(eventType, async ({ body, ack }) => {
    // Ack immediately (within 3 seconds)
    await ack();
    // Route asynchronously
    routeEvent(body).catch(err => console.error('[router] Route error:', err));
  });
}

client.on('connected', () => {
  console.log('[router] Socket Mode connected');
  console.log(`[router] Routes: ${Object.keys(config.channels).length} channels, ${Object.keys(config.members).length} members`);
});

client.on('disconnected', () => {
  console.log('[router] Socket Mode disconnected — will reconnect');
});

// Start
console.log('[router] Starting Slack event router...');
client.start().catch(err => {
  console.error('[router] Fatal:', err);
  process.exit(1);
});
```

Note: The `@slack/socket-mode` library handles reconnection, ping/pong, and the `apps.connections.open` call automatically. We don't need to implement the Socket Mode protocol ourselves — the SDK does it.

### 2. Routing Config (`/opt/conga/config/routing.json`)

Generated by user-data at bootstrap:

```json
{
  "channels": {
    "CEXAMPLE01": "http://conga-UEXAMPLE01:18789/slack/events",
    "CEXAMPLE02": "http://conga-UEXAMPLE02:18789/slack/events"
  },
  "members": {
    "UEXAMPLE01": "http://conga-UEXAMPLE01:18789/slack/events",
    "UEXAMPLE02": "http://conga-UEXAMPLE02:18789/slack/events"
  }
}
```

Container hostnames resolve via Docker's built-in DNS on each per-user network (the container name is the hostname).

### 3. Container Config Changes

Each container's `openclaw.json` changes from:

```json
{
  "channels": {
    "slack": {
      "mode": "socket",
      "enabled": true,
      ...
    }
  }
}
```

To:

```json
{
  "channels": {
    "slack": {
      "mode": "http",
      "enabled": true,
      "webhookPath": "/slack/events",
      "userTokenReadOnly": true,
      "groupPolicy": "allowlist",
      "streaming": "partial",
      "nativeStreaming": true,
      "channels": {
        "<CHANNEL_ID>": { "allow": true, "requireMention": false }
      }
    }
  }
}
```

Key changes:
- `mode` → `"http"`
- `webhookPath` added
- `appToken` no longer needed (router holds it)
- `botToken` still needed (container sends replies via Web API)
- `signingSecret` needed if Conga Line verifies webhook signatures in HTTP mode

### 4. Env Var Changes

**Router container env:**
- `SLACK_APP_TOKEN` — moved from user containers to router
- `SLACK_BOT_TOKEN` — router may need this for connection metadata
- `ROUTER_CONFIG=/opt/conga/config/routing.json`

**User container env:**
- Remove `SLACK_APP_TOKEN`
- Keep `SLACK_BOT_TOKEN` (for sending replies)
- Add `SLACK_SIGNING_SECRET` (if needed for HTTP mode verification)
- Keep all per-user secrets (ANTHROPIC_API_KEY, etc.)

### 5. New Shared Secret

Add to Secrets Manager:
- `conga/shared/slack-signing-secret` — the Slack app's signing secret (from app settings → Basic Information → Signing Secret)

### 6. Docker Networking

Router joins every per-user network:

```bash
# In user-data, after creating per-user networks and starting the router:
docker network connect conga-UEXAMPLE01 conga-router
docker network connect conga-UEXAMPLE02 conga-router
```

Container names serve as DNS hostnames within each network. The router resolves `conga-UEXAMPLE01` on the `conga-UEXAMPLE01` network and `conga-UEXAMPLE02` on the `conga-UEXAMPLE02` network.

## Edge Cases

| Scenario | Handling |
|---|---|
| Event with no channel (app_home, etc.) | Fall back to user-based routing; drop if no match |
| Router restarts | `@slack/socket-mode` auto-reconnects; systemd restarts on crash; ~5s downtime |
| Container is down when event arrives | HTTP POST fails; router logs error; Slack event is lost. Acceptable — user retries. |
| New user added | Regenerate routing.json, restart router, connect to new network |
| Slack sends event for unknown channel | Router logs and drops — no crash |
| Multiple events in rapid succession | Node.js event loop handles async; ack is instant, forwarding is fire-and-forget |
| Bot's own messages | Slack sends bot's own messages as events; Conga Line handles this (ignores self) |
| Thread replies | Thread events include `channel` field — routes correctly |
| Reactions, file uploads, etc. | All contain `channel` or `item.channel` — routing logic handles these |

## Validation Steps

1. Build router, verify `npm install` succeeds
2. Start router container with routing config
3. Verify router connects to Slack: `[router] Socket Mode connected`
4. Send message in channel CEXAMPLE01 → verify it reaches container UEXAMPLE01
5. Send message in channel CEXAMPLE02 → verify it reaches container UEXAMPLE02
6. Send DM to the bot → verify it reaches the sender's container
7. Verify streaming works: response updates in real-time in Slack
8. Verify container isolation: containers cannot reach each other's networks
9. Kill router → verify systemd restarts it → verify reconnection
